# femtoClaw v0.9.0 Execution Audit (실행 감사 리포트)

## 개요
이 문서는 v0.9.0까지 구현된 주요 기능들이 `spec.md` 명세대로 정상 작동하는지와 에지 케이스/예외 상황에 대해 어떻게 대응하는지를 검증한 실행 감사 결과입니다.

## 1. 오프라인 큐잉 (Offline Queuing)

### 정상 케이스 (스펙 일치 여부)
- **시나리오**: 사용자가 정상적인 텔레그램 메시지나 TUI 메시지를 전송하고, 외부 LLM API가 성공적으로 200 OK를 반환.
- **결과**: `handle_message`가 성공적으로 응답을 반환하며 기록(`history`, `db`, `session`)이 모두 정상. `pending_queue`는 비어 있음.

### 에지 케이스 (실패 및 영속성)
- **에지 케이스 시나리오**: 잘못된 API Endpoint 주입 또는 네트워크 단절 상태로 LLM 요청 실패 강제 유도.
- **기대 동작**: LLM API 에러(Timeout/Unknown Host 등) 발생 시 프로그램 패닉 없이, 사용자 메시지는 히스토리에서 제거되고 `pending_queue`에 삽입되어야 함. 세션이 다시 열릴 디스크(`pending_queue.json`)에서 이를 불러와야 함.
- **실제 동작 (Pass ✅)**: `tests/offline_queueing.rs` 통합 테스트를 통해 검증 완료.
  - 잘못된 API 설정 시 패닉 없이 `⚠️` 에러 메시지와 현재 대기 큐 크기를 반환함.
  - `pending_queue.json` 파일에 사용자 메시지가 정상 기록됨("영속성"). 
  - 앱 재구동/워커 재생성 시 `ChatSession::new()`에서 큐를 파일에서 읽어 복원함.
  - LLM 연결 성공 시 `drain_pending_queue()`를 통해 큐에 보관된 메시지가 순차적으로 자동 전송 처리됨.

## 2. 텔레그램 메세지 분할 전송 (Chunking)

### 정상 케이스
- **시나리오**: 에이전트가 4096자 제한 이내의 일반 텍스트 응답 전송.
- **결과**: 텔레그램 봇 API 내에서 한 번의 `send_message`로 전송 완료.

### 에지 케이스 (초장문 응답/제한 초과)
- **에지 케이스 시나리오**: 도구 반환값이나 LLM 오류로 5,000자 이상의 매우 긴 텍스트가 텔레그램 콜백으로 반환됨.
- **기대 동작**: 텔레그램 API에서 `MessageTooLong` 에러를 반환하며 봇이 작동 중지되는 대신 4000자 단위로 자동 분할되어 전송되어야 함.
- **실제 동작 (Pass ✅)**:
  - `telegram.rs:309`에 적용된 `chars().chunks(4000)` 로직을 통해 4000개의 UTF-8 문자를 넘으면 독립된 `send_message` 코루틴을 순차적으로 호출하여 여러 개의 말풍선으로 쪼개어 보냄.

## 3. DB 커넥션 재사용 (Connection Caching)

### 정상 케이스
- **시나리오**: `set_db_path()` 수행 시, 커넥션을 한 번 맺고 활성화된 세션 내내 유지하며 `insert_action` 수행.
- **결과**: 매 메시지(턴)마다 파일을 open/close 하는 I/O 오버헤드가 제거되어 빠른 속도 달성. `tests/simulation_30turn.rs` 30턴 시뮬레이션에서 무결점 통과 및 I/O 시간 대폭 감소 확인.

### 에지 케이스 (Locks / Concurrency)
- **에지 케이스 시나리오**: `sqlite_busy` (파일이 다른 프로세스나 백업 크론 데몬에 의해 락 걸린 경우).
- **기대 동작**: 락 에러 발생 시 SQLite 오류가 전체 애플리케이션의 패닉으로 전파되지 않아야 함. 기록을 1회 스킵할지언정 대화 세션은 무너지면 안 됨.
- **실제 동작 (Pass ✅)**: `insert_action`과 `FemtoDb::open` 모두 `Result::Err`을 반환하며, `chat_loop.rs:289`에서 `let _ = db.insert_action(...)`으로 에러를 묵시적으로 무시하고 대화를 지속하도록 처리됨 (패닉 없음).

## 결론
v0.9 로드맵에 반영된 기능들은 명세(`spec.md`)대로 완성도 높게 작동하며, 
특히 비정상 환경(LLM 실패, 초장문 출력, DB 락)에서 다운타임이나 데이터 소실을 방지하는 **단단한 복원력(Resilience)**이 확인되었습니다.
